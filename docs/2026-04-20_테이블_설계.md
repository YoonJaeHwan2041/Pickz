# Pickz 테이블 설계 초안

작성일: 2026-04-20

## 문서 목적

- MVP 범위 기준으로 핵심 테이블 구조를 정의한다.
- 각 컬럼의 의미와 설계 이유를 함께 기록해 의사결정 근거를 남긴다.
- 인덱스는 초기 단계에서 제외하고, 제약은 유니크(UNIQUE)만 최소 적용한다.
- 이후 ERD, JPA 엔티티, API 명세의 기준 문서로 사용한다.

---

## 1) `users`

### 역할

- 모든 계정(일반 유저, 업체 유저, 관리자)의 공통 인증/인가 정보를 관리한다.

### 컬럼 정의 (컬럼별 주석)

- `id` (BIGINT, PK, AUTO_INCREMENT) : 사용자 고유 식별자
- `email` (VARCHAR, UNIQUE) : 로그인 아이디(이메일), 중복 불가
- `password_hash` (VARCHAR) : 비밀번호 해시값
- `nickname` (VARCHAR, UNIQUE) : 서비스 노출 이름, 중복 불가
- `user_type` (ENUM: GENERAL, BUSINESS, ADMIN) : 계정 유형
- `status` (ENUM: ACTIVE, BLOCKED, DELETED) : 계정 상태
- `profile_image_url` (VARCHAR, NULL) : 사용자 프로필 이미지 URL
- `created_at` (DATETIME) : 생성 시각
- `updated_at` (DATETIME) : 수정 시각

### 제약 조건 (UNIQUE만 적용)

- UNIQUE: `email`
- UNIQUE: `nickname`

### 설계 이유

- 로그인은 `users` 단일 테이블만 조회하면 되도록 공통 계정을 한곳에 모은다.
- `user_type`으로 일반/업체/관리자 분기만 처리하면 되므로 인증 흐름이 단순해진다.
- 프로필 이미지는 공통 수요이므로 `users`에 기본 컬럼으로 둔다.

---

## 2) `seller_profiles`

### 역할

- 판매 기능을 사용하는 사용자의 공통 판매자 정보를 관리한다.

### 컬럼 정의 (컬럼별 주석)

- `id` (BIGINT, PK, AUTO_INCREMENT) : 판매자 프로필 식별자
- `user_id` (BIGINT, UNIQUE) : `users.id`와 1:1 매핑
- `seller_display_name` (VARCHAR) : 판매자 노출 이름
- `seller_status` (ENUM: PENDING, ACTIVE, SUSPENDED) : 판매 가능 상태
- `created_at` (DATETIME) : 생성 시각
- `updated_at` (DATETIME) : 수정 시각

### 제약 조건 (UNIQUE만 적용)

- UNIQUE: `user_id`

### 설계 이유

- 일반 유저가 향후 판매자로 전환될 수 있으므로 판매자 정보를 분리 저장한다.
- `users`는 인증 중심, `seller_profiles`는 판매 운영 중심으로 책임을 나눈다.

---

## 3) `business_profiles`

### 역할

- 업체 유저 전용 추가 정보를 관리한다.

### 컬럼 정의 (컬럼별 주석)

- `id` (BIGINT, PK, AUTO_INCREMENT) : 업체 프로필 식별자
- `seller_profile_id` (BIGINT, UNIQUE) : `seller_profiles.id`와 1:1 매핑
- `company_name` (VARCHAR) : 업체명
- `business_number` (VARCHAR, UNIQUE) : 사업자등록번호, 중복 불가
- `owner_name` (VARCHAR) : 대표자명
- `business_image_url` (VARCHAR, NULL) : 업체 로고/대표 이미지 URL
- `created_at` (DATETIME) : 생성 시각
- `updated_at` (DATETIME) : 수정 시각

### 제약 조건 (UNIQUE만 적용)

- UNIQUE: `seller_profile_id`
- UNIQUE: `business_number`

### 설계 이유

- 일반 판매자와 업체 판매자의 데이터 요구사항이 달라 업체 전용 테이블을 분리한다.
- 관리자/일반 유저에 불필요한 업체 컬럼이 `users`에 섞이지 않도록 설계한다.

---

## 4) `auctions`

### 역할

- 경매 상품 정보와 경매 진행 상태를 관리한다.

### 컬럼 정의 (컬럼별 주석)

- `id` (BIGINT, PK, AUTO_INCREMENT) : 경매 식별자
- `seller_id` (BIGINT) : 판매자(`users.id`)
- `title` (VARCHAR) : 경매 제목
- `description` (TEXT, NULL) : 상품 설명
- `start_price` (DECIMAL) : 시작가
- `bid_unit` (DECIMAL) : 최소 입찰 단위
- `current_price` (DECIMAL) : 현재 최고 입찰가
- `start_at` (DATETIME) : 경매 시작 시각
- `end_at` (DATETIME) : 현재 종료 시각(연장되며 변경)
- `max_end_at` (DATETIME, NULL) : 최대 연장 가능 시각(NULL이면 상한 미사용)
- `extension_window_sec` (INT) : 종료 임박 판단 구간(N초)
- `extension_sec` (INT) : 1회 연장 시간(M초)
- `status` (ENUM: SCHEDULED, LIVE, ENDED, CANCELLED) : 경매 상태
- `version` (BIGINT) : 낙관적 락 버전 값
- `created_at` (DATETIME) : 생성 시각
- `updated_at` (DATETIME) : 수정 시각

### 제약 조건 (UNIQUE만 적용)

- 없음

### 설계 이유

- `current_price`를 별도 보관해 목록/상세 조회 시 계산 비용을 줄인다.
- `max_end_at`을 NULL 허용으로 두어 "최대 연장 시간 미사용" 정책도 지원한다.
- `version` 컬럼으로 낙관적 락 충돌 감지 기반을 만든다.

---

## 5) `auction_images`

### 역할

- 경매 상품의 대표 이미지/상세 이미지를 관리한다.

### 컬럼 정의 (컬럼별 주석)

- `id` (BIGINT, PK, AUTO_INCREMENT) : 이미지 식별자
- `auction_id` (BIGINT) : 대상 경매(`auctions.id`)
- `image_url` (VARCHAR) : 이미지 URL
- `image_type` (ENUM: THUMBNAIL, DETAIL) : 대표/상세 이미지 구분
- `sort_order` (INT) : 상세 이미지 노출 순서
- `created_at` (DATETIME) : 생성 시각

### 제약 조건 (UNIQUE만 적용)

- 없음

### 설계 이유

- 이미지가 여러 장이므로 경매 테이블에서 분리해 관리해야 확장성이 좋다.
- 썸네일/상세를 명시적으로 분리해 프론트 표시 로직을 단순화한다.

---

## 6) `bids`

### 역할

- 입찰 이력을 저장하는 정본 데이터 테이블이다.

### 컬럼 정의 (컬럼별 주석)

- `id` (BIGINT, PK, AUTO_INCREMENT) : 입찰 식별자
- `auction_id` (BIGINT) : 대상 경매(`auctions.id`)
- `bidder_id` (BIGINT) : 입찰 사용자(`users.id`)
- `amount` (DECIMAL) : 입찰 금액
- `bid_status` (ENUM: ACCEPTED, REJECTED) : 입찰 처리 결과
- `reject_reason` (VARCHAR, NULL) : 실패 시 사유
- `requested_at` (DATETIME) : 클라이언트 요청 시각
- `created_at` (DATETIME) : 서버 저장 시각

### 제약 조건 (UNIQUE만 적용)

- 없음

### 설계 이유

- 입찰 로그를 애플리케이션 로그만으로 처리하지 않고 DB에 남겨 정합성과 추적성을 확보한다.
- 분쟁 대응, 낙찰 근거, 사후 분석을 위해 입찰 이력은 영속 저장이 필요하다.

---

## 7) `auction_results`

### 역할

- 경매 종료 결과를 별도로 확정 저장한다.

### 컬럼 정의 (컬럼별 주석)

- `id` (BIGINT, PK, AUTO_INCREMENT) : 결과 식별자
- `auction_id` (BIGINT, UNIQUE) : 결과 대상 경매(`auctions.id`), 경매당 1건
- `winner_id` (BIGINT, NULL) : 낙찰자(`users.id`), 유찰이면 NULL
- `winning_bid_id` (BIGINT, NULL) : 최종 낙찰 입찰(`bids.id`), 유찰이면 NULL
- `final_price` (DECIMAL, NULL) : 최종 낙찰가, 유찰이면 NULL
- `result_status` (ENUM: SOLD, UNSOLD, CANCELLED) : 결과 상태
- `closed_at` (DATETIME) : 경매 종료 시각
- `created_at` (DATETIME) : 결과 생성 시각

### 제약 조건 (UNIQUE만 적용)

- UNIQUE: `auction_id`

### 설계 이유

- `auctions`와 종료 결과 책임을 분리해 유지보수성을 높인다.
- 경매 상세 조회에서는 조인으로 결과를 가져오고, 정산/통계는 결과 테이블 중심으로 조회한다.
- `auction_id` 유니크로 중복 종료 결과를 DB 레벨에서 방지한다.

---

## 8) `refresh_tokens`

### 역할

- JWT 재발급 토큰을 저장/폐기 관리한다.

### 컬럼 정의 (컬럼별 주석)

- `id` (BIGINT, PK, AUTO_INCREMENT) : 토큰 식별자
- `user_id` (BIGINT) : 사용자(`users.id`)
- `token_hash` (VARCHAR, UNIQUE) : 리프레시 토큰 해시값
- `expires_at` (DATETIME) : 만료 시각
- `revoked_at` (DATETIME, NULL) : 강제 만료 시각
- `created_at` (DATETIME) : 생성 시각

### 제약 조건 (UNIQUE만 적용)

- UNIQUE: `token_hash`

### 설계 이유

- 원문 토큰 저장 대신 해시 저장으로 보안 리스크를 낮춘다.
- 토큰 중복 저장 방지를 위해 `token_hash` 유니크를 둔다.

---

## 테이블 간 관계 요약

- `users(1) - (0..1) seller_profiles` : 일반 유저는 없을 수 있고, 판매자는 1개 보유
- `seller_profiles(1) - (0..1) business_profiles` : 업체 판매자만 업체 프로필 보유
- `users(1) - (N) auctions` : 한 판매자가 여러 경매 등록
- `auctions(1) - (N) auction_images` : 한 경매에 여러 이미지 등록
- `users(1) - (N) bids` : 한 사용자가 여러 입찰 수행
- `auctions(1) - (N) bids` : 한 경매에 여러 입찰 발생
- `auctions(1) - (0..1) auction_results` : 종료 시 결과 1건 확정

---

## 동시성 전략 메모 (낙관적/비관적 논의 결과)

- 초기 구현은 `낙관적 락` 기반으로 진행한다.
  - 이유: 처리량/응답속도 우선, 구현 복잡도와 운영 난이도 균형
  - 방식: `auctions.version`을 사용해 충돌 감지 후 재시도 또는 실패 응답 처리
- 테스트 단계에서 경합 비율과 실패율을 측정한다.
  - 부하 테스트에서 충돌이 과도하면 `비관적 락`(또는 Redis 분산 락) 전환을 검토한다.
- 정리: `초기에는 낙관적 락으로 빠르게 검증하고, 테스트 결과에 따라 비관적 락을 실험/적용한다.`

---

## 인덱스/제약 운영 원칙 (현재 합의)

- 인덱스는 초기 단계에서 생성하지 않는다.
- 제약 조건은 UNIQUE만 최소 적용한다.
- 운영/테스트 데이터로 병목 지점이 확인되면 이후 단계에서 인덱스를 점진적으로 추가한다.

# Bid Domain Entity Decision

작성일: 2026-04-20

## 도메인 책임

- 입찰 요청 이력과 처리 결과를 정본 데이터로 저장한다.
- 경매 낙찰 확정의 근거 데이터를 보존한다.

## 엔티티 구성

### 1) `bids`

- `id`: 입찰 식별자(PK)
- `auction_id`: 대상 경매 식별자
- `bidder_id`: 입찰 사용자 식별자
- `amount`: 입찰 금액
- `bid_status`: 처리 결과(ACCEPTED, REJECTED)
- `reject_reason`: 실패 사유(NULL 가능)
- `requested_at`: 요청 발생 시각
- `created_at`: 저장 시각

## 왜 이렇게 설계했는가

- 입찰은 로그 성격이 강하지만, 로그 파일만으로는 분쟁/감사/낙찰 근거 보장이 어렵다.
- 따라서 입찰 이력은 DB에 정본으로 저장하고, 로그 시스템은 보조 추적 용도로 분리한다.
- 성공/실패를 모두 저장해 트러블슈팅과 운영 분석에 활용한다.

## 제약/인덱스 정책

- 현재 단계는 UNIQUE 제약을 별도로 두지 않는다.
- 일반 인덱스는 초기 단계에서 제외하고, 조회 병목 구간이 확인되면 추가한다.

## 동시성/Redis 메모

- 입찰 수용 여부 판단의 주체는 경매 도메인 정책과 락 전략이다.
- Redis는 실시간 반영 보조에 활용할 수 있지만, 입찰 이력 정본 저장소는 DB를 유지한다.
